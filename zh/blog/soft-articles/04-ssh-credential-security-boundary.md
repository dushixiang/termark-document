# SSH 客户端保存密码和密钥，安全边界到底在哪里

只要一个 SSH 客户端支持保存密码、私钥、跳板机配置，就一定会遇到一个问题：

用户凭什么相信你？

这个问题很合理。

服务器密码和私钥不是普通配置。它们一旦泄露，影响的可能不是一个账号，而是一批服务器、数据库、内网服务，甚至生产环境。

所以“放心，我们很安全”这种话没有意义。

更重要的是把边界讲清楚：数据存在哪里，怎么加密，密钥在哪里，同步时服务端能不能看到明文，密码丢了能不能恢复。

## 保存凭证本身就是一个敏感动作

很多开发者一开始会说：“我不保存密码，只用系统 SSH。”

这是一个很干净的选择。

但现实里，很多人最后还是会让工具保存凭证。原因也很简单：

- 机器太多
- 账号太多
- 私钥太多
- 跳板机配置复杂
- 公司电脑和个人电脑都要用
- 客户环境经常切换
- 有些设备仍然只支持密码登录

不保存凭证当然更简单，但对高频使用者来说，体验成本会很高。

所以问题不是“SSH 客户端该不该保存凭证”。

更实际的问题是：如果要保存，至少应该做到什么程度？

## 第一件事：不要明文落盘

这是最低要求。

如果一个工具把服务器密码、私钥口令、代理密码直接明文写进数据库或配置文件，那风险就太高了。

本机被其他程序读取、备份工具上传、误打包、误同步、磁盘被拷走，都可能导致凭证泄露。

用户未必会检查工具的数据目录。

但工具作者应该默认把敏感字段当成不能明文落盘的数据。

这里的“敏感字段”不只是 SSH 密码，还包括：

- 私钥内容
- 私钥口令
- 代理密码
- 同步密码
- 跳板机凭证
- API Key
- 其他连接密钥

资产名称、分组、主机地址是否算敏感，要看具体场景。有些公司连主机地址都不希望暴露，但至少密码和密钥一定应该加密。

## 第二件事：加密密钥不能和密文随便放在一起

“我们加密保存了”也不够。

如果加密密钥和密文放在同一个普通配置文件里，只是把明文换了一种形式，实际安全收益很有限。

更合理的做法是利用操作系统提供的安全存储能力，比如 macOS Keychain、Windows Credential Manager 之类。

工具本地数据库里保存密文，解密需要从系统安全存储里拿到本地数据密钥。

这样至少避免了“复制一个数据库文件就拿到全部明文”的情况。

当然，这也不是绝对安全。

如果用户的本机已经被完全控制，攻击者总有办法在运行时拿到数据。但安全设计不是追求一句“绝对安全”，而是尽量减少低成本泄露路径。

## 第三件事：便携版要单独考虑

Windows 用户经常喜欢便携版。

便携版的好处是方便移动，不需要完整安装。问题是，便携版的数据也更容易被复制到别的机器上。

这时候如果仍然完全依赖当前系统的 keychain，迁移体验会变差；如果完全不加密，又不安全。

比较合理的方式是让便携版有自己的本地加密口令。

用户第一次启动时设置一个口令，用它派生本地数据密钥。数据库里只保存必要的非秘密参数，比如 salt、KDF 参数等，不保存明文口令，也不保存明文数据密钥。

迁移到另一台电脑时，用户带走完整数据目录，并重新输入本地加密口令解锁。

这会多一步操作，但边界比较清楚：能不能解密，取决于用户是否掌握这个口令。

## 第四件事：同步前应该在客户端加密

多设备同步是另一个敏感点。

很多人希望公司电脑、家里电脑、笔记本之间共享 SSH 资产和命令片段。否则每换一台机器，都要重新录入一遍主机、凭证、跳板机、端口转发规则。

但同步意味着数据要离开本机。

这时候要问几个问题：

- 上传到服务端的是明文还是密文
- 服务端能不能解密
- 同步密码会不会发给服务端
- 使用 WebDAV、S3、iCloud、本地目录时是否同样加密
- 多设备同时修改时怎么处理冲突
- 密码丢失后能不能恢复

我个人更倾向于客户端加密后再上传。

也就是说，用户设置同步密码，客户端用它派生密钥，把需要同步的数据加密后再发出去。服务端只保存密文，不拿同步密码，也不具备解密能力。

这样做的好处是边界清楚。

服务端即使被读取，也不应该直接看到用户的服务器凭证。

代价是：同步密码丢了，服务端也帮不了你。

这不是一个“体验更好”的设计，但它是一个边界更干净的设计。

## 第五件事：不能为了“找回密码”牺牲安全边界

很多产品会提供找回密码。

对普通账号来说，这是合理的。

但如果一个工具声称同步数据是端到端加密，同时又能帮你找回同步密码、恢复所有明文凭证，那就要小心了。

这两件事往往是冲突的。

如果服务端能帮你恢复明文，通常意味着服务端掌握某种解密能力，或者至少可以绕过你的同步密码。

对 SSH 凭证这种数据，我更愿意接受一个“不那么贴心”的结果：

密码丢了，就无法恢复。

这听起来不舒服，但它说明服务端没有替你解密的后门。

用户需要自己备份同步密码，工具需要把这个风险提前说清楚。

安全边界不能靠事后解释。

## 第六件事：同步冲突不能静默覆盖

多设备同步还有一个常被忽视的问题：冲突。

比如你在公司电脑改了一个主机配置，又在家里电脑改了同一批资产。如果工具静默覆盖，就可能出现：

- 新增资产丢失
- 凭证被旧版本覆盖
- 分组结构回退
- 命令片段消失
- 端口转发规则被替换

对普通文档来说，冲突已经很烦。

对服务器凭证和资产来说，冲突更麻烦。

所以同步系统至少应该在检测到版本冲突时告诉用户，而不是悄悄用一端覆盖另一端。

哪怕处理方式只是“从云端恢复”或“用本地覆盖云端”，也比无声覆盖好。

## 第七件事：安全说明要写给普通用户看

安全设计不应该只存在于代码里。

用户不一定会读源码，也不一定懂每个算法细节，但他至少应该能看懂这些问题：

- 我的密码是不是明文保存在本地
- 本地加密密钥保存在哪里
- 换电脑后为什么需要重新解锁
- 同步数据上传前有没有加密
- 服务端能不能看到我的密码和私钥
- 同步密码丢了怎么办
- 删除系统 keychain 后会发生什么

这些说明越具体，越能建立信任。

不要只写“银行级加密”“军工级安全”。

这种词看起来很强，实际上没有边界。

真正有用的是告诉用户：哪里加密，谁能解密，丢了什么会造成什么后果。

## 我做 Termark 时的安全取舍

Termark 是一个桌面端 SSH 客户端，会涉及主机、凭证、跳板、同步配置等本地数据。

所以我在设计时把几个边界定得比较清楚：

- 本地敏感字段不明文落盘
- 普通安装版使用系统 keychain 保存本地数据密钥
- Windows 便携版使用独立的本地加密口令模式
- 同步数据在客户端加密后再上传
- 同步密码不发给服务端
- 服务端保存的是密文，不负责解密
- 同步密码丢失后，服务端无法帮用户恢复明文数据

这不是说它可以抵抗所有攻击。

任何本地工具都要承认一个前提：如果用户设备本身已经被完全控制，运行时数据就很难绝对安全。

但工具至少应该避免低级风险：不要明文存凭证，不要把同步数据明文上传，不要让服务端默认拥有解密用户服务器信息的能力。

## 保存凭证前，可以问这几个问题

如果你正在选择 SSH 客户端，或者准备把服务器凭证放进某个工具里，可以先问：

- 它会不会明文保存密码和私钥
- 本地加密密钥放在哪里
- 是否使用系统 keychain 或类似机制
- 便携版怎么处理本地加密
- 同步数据上传前是否加密
- 同步密码是否发给服务端
- 服务端能不能解密用户数据
- 同步密码丢失后是否还能恢复
- 多设备冲突是否会静默覆盖
- 安全说明是否足够具体

这些问题不一定都有完美答案。

但一个工具至少应该愿意正面回答。

## 最后

SSH 客户端保存密码和密钥，不是一个可以用“放心”带过的问题。

用户真正需要知道的是边界：本地怎么存，云端怎么同步，谁能解密，丢了什么会造成什么后果。

我做 Termark 时更愿意选择一个清楚但不那么讨好的边界：敏感数据本地加密，同步前客户端加密，服务端不拿同步密码，也不能帮用户解密。

这意味着用户要自己保管好同步密码。

但对服务器凭证来说，我认为这是更合理的取舍。
