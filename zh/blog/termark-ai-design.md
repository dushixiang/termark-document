# Termark AI 助手设计：我为什么没有做一个“全自动运维 Agent”

在我另一款产品 NextTerminal 里，我很早就做过一个 AI 助手。

那时候的设计很简单：用户在后台配置 API、模型和提示词，打开终端后可以在旁边问一句“这个命令怎么写”。AI 输出命令，用户看一眼，再决定要不要执行。

功能并不复杂，但反馈还不错。

因为它解决的是一个很具体的问题：人在终端里工作，经常不是完全不会，而是需要一个能快速给出方向的助手。比如查日志、看进程、写一条 `grep`、解释一段报错、补一个自己一时想不起来的参数。

但我一直没有把它继续做成更激进的“自动运维 Agent”。原因也很简单：服务器不是代码仓库。

代码仓库里 AI 删错了文件，大多数时候还能从 git 里找回来。Linux 机器上执行错一条命令，删掉的可能是日志、配置、数据库文件，甚至是一台正在跑业务的机器。

所以 Termark 里的 AI 不是按“让 AI 尽可能多做事”设计的，而是按“让 AI 在可控边界内提高效率”设计的。

---

## 从一问一答，到真正站在终端旁边

我现在已经离不开 Claude Code、Codex 这类编程 Agent 了。一个账号的额度不够用，我也会在不同渠道之间切换。

用久了以后，再回头看传统的“终端旁边放一个聊天框”，就会觉得不够了。

因为真实的排查过程不是：

> 用户问一句，AI 答一句。

而是：

> 看最近终端输出，判断下一步该查什么，执行一条只读命令，根据结果继续分析，必要时再给出修改建议。

所以 Termark 的 AI 助手也必须是 Agent 形态。

但这个 Agent 不应该一上来就拥有一堆复杂能力。Termark 最核心的现场就是用户当前可见的 SSH 终端，所以工具边界也收敛在这个终端会话里：

- 查看当前终端环境；
- 在当前终端里执行命令；
- 通过受控命令查看目录、读取文件、搜索内容；
- 在 Codex / Claude 的 MCP 路径里，进一步拆出目录查看、文件读取、文件搜索和文件写入等工具。

这些工具最后都落到当前 SSH 会话上，而不是让 AI 自己绕过 Termark 去连服务器。

这样做有三个好处。

第一，AI 看到和操作的是用户当前正在看的那台机器，不会莫名其妙跑到别的环境。

第二，命令会出现在当前终端里，输出、退出码、耗时、超时状态都会回传。用户不是在一个黑盒里等 Agent 操作完成，而是能看到它到底做了什么。

第三，也是我觉得最重要的一点：它不是一个后台 shell。

很多 Agent 执行命令时，是在自己开的后台会话里跑。用户看到的是工具结果，不一定看得到真实终端现场。

Termark 不是这样。

Termark 的 AI 绑定的是当前可见终端。你在终端里 `su` 到另一个用户，AI 后面执行的命令就在这个用户上下文里跑；你切到某个目录，后续命令也会在这个现场继续执行；命令如果要求输入数据库密码、二次确认、交互式选择，用户可以直接在终端里接着输入。

所以它更像是你旁边的人把命令敲进当前终端，而不是一个躲在后台的自动化脚本。

这会牺牲一点“纯自动化”的感觉，但换来的是现场一致性和可接管性。

## 为什么我没有默认放开所有命令

很多人用 AI Agent 的时候，最烦的是每一步都确认。

我理解这个感受。写代码时频繁确认 `ls`、`cat`、`rg` 确实很打断。

但服务器场景不一样。

Termark 面对的是 SSH 资产，里面可能是个人 VPS，也可能是生产环境。AI 说“我清理一下临时文件”，背后可能就是一条 `rm -rf`。AI 说“我重启服务看看”，背后可能影响线上流量。

所以我的策略是：

**默认只自动放行明确只读、可观察的命令；改变远端状态、无法证明安全、或者解析不清楚的命令，都必须让用户确认。**

代码里有一套命令风险判断逻辑，会拆 shell token、识别管道、重定向、子命令、反引号、命令替换等情况。只要命令里出现写入、删除、移动、安装、重启、权限变更、输出重定向等可能改变状态的动作，就会进入确认流程。

设置里也保留了更保守的模式：所有工具调用都确认。

我没有做“开发者模式：永不确认”。

原因不是我不相信 AI，而是我不相信每个人接入的模型都足够聪明。Termark 支持 OpenAI 兼容接口，用户可能接 OpenAI、DeepSeek、Qwen、Kimi、Ollama，也可能接各种中转站和本地模型。模型能力、工具调用质量、上下文理解能力都不统一。

如果产品给了一个“永不确认”的开关，最后出问题的时候，损失发生在用户的服务器上。

这不是我想要的默认设计。

## 内置 Agent：轻量、可控、够用

Termark 内置了一条 OpenAI 兼容的 Agent 路径。

你可以配置不同的 API Profile，比如 OpenAI、DeepSeek、OpenRouter、Qwen、Kimi、Ollama 或自定义接口。每个 Profile 可以配置：

- API 地址；
- API Key；
- 模型列表；
- 当前模型；
- reasoning / thinking 相关参数；
- 最大重试次数；
- 自定义 User-Agent。

我自己测试时，也会用响应快、成本可控的模型做高频终端辅助。因为 Termark 给模型的上下文很克制：最近若干行终端输出、当前会话范围、必要的系统提示词，以及用户这次的问题。

它不是把整个服务器状态塞给模型，也不是把所有历史无脑塞满。

这带来的结果是：日常解释日志、生成排查命令、看配置片段、搜索文件，不需要一开始就堆很大的上下文。

当然，轻量模型也有边界。

所以 Termark 对内置 Agent 的定位不是“替你全自动完成运维任务”，而是“在终端旁边给你一个能看现场、能有限执行命令的助手”。

你要它查磁盘、看端口、分析报错，它很适合。

你要它不经确认改生产配置，我不鼓励。

## 如果你更信任 Codex 或 Claude

还有一种用户很明确：

> 我不想用你内置的 Agent，我已经习惯 Codex / Claude Code 了。

这个需求也合理。

所以 Termark 不是只做一个自己的聊天框，而是把 Codex 和 Claude Code 也接进来了。

Codex 这边走的是 `codex-acp`。Termark 启动 `codex-acp` 进程，通过 ACP 会话发送 prompt，并把 Termark 自己的 MCP 工具注册给它。这样 Codex 不是凭空操作，而是通过 Termark 提供的受控工具访问当前 SSH 会话。

Claude Code 这边走的是 CLI。Termark 会为当前会话生成 MCP 配置，启动 `claude -p` 的 stream-json 输出模式，并且只允许 Termark 暴露的 MCP 工具。Claude 自带的 `Bash`、`Read`、`Write`、`Edit`、`WebSearch` 等工具会被禁用。

这点很重要。

我不是把 Claude Code 放进 Termark 后就让它随便跑本地命令，而是让它通过 Termark 的 MCP 工具访问远端终端。这样确认策略、会话范围、历史记录、工具结果展示都还能收回来。

用户可以继续使用自己信任的 Agent 能力，Termark 负责把它放进一个适合 SSH 场景的边界里。

## 历史记录不是简单聊天记录

AI 助手如果只保存问答，其实价值有限。

Termark 保存的是会话级历史：

- 用户输入；
- AI 回复；
- 工具调用；
- 工具参数；
- 工具结果；
- 错误信息；
- token 使用量；
- 外部 Agent 的 session id。

历史范围也不是全局混在一起，而是尽量跟机器绑定。能识别到资产目标时，就按机器维度组织；识别不到时，再退回到终端会话维度。

这意味着你在一台机器上排查过的问题，下次打开同一台机器，还能接着之前的 AI 会话看。

对 Codex / Claude 这类外部 Agent，Termark 也会记录外部 session id。能恢复就恢复，不能恢复再创建新会话。

这个设计比“聊天框历史”多做了一步：它承认 AI 操作是排查现场的一部分，而不是孤立的问答。

## 最近终端输出是关键上下文

我不希望用户每次都手动复制一段终端输出给 AI。

所以 Termark 会从当前终端抓取最近 N 行输出，作为上下文传给 Agent。这个 N 可以在设置里调整。

这样你刚执行完：

```bash
systemctl status nginx
```

然后问：

```text
帮我看一下为什么没启动
```

AI 就能结合刚才的输出分析，而不是反问你“请提供错误日志”。

但这里也没有过度设计。

Termark 没有试图把整个终端滚动缓冲区、所有文件、整台机器状态都塞给模型。上下文越大，成本越高，噪音也越多。对终端辅助来说，最近输出通常已经足够覆盖大部分问题。

## 外部 CLI：给你本地已有的 Agent 用

还有一类场景，和 Termark 内置 AI 正好反过来。

有些人已经在本地终端里重度使用 Codex、Claude Code 或 OpenCode，不想切到 Termark 的 AI 面板里。

那 Termark 就提供外部 CLI。

在设置页里点几下，就可以把 `termark` 命令安装到 PATH，并把 Termark skill 安装到 Codex、Claude、OpenCode 的 skills 目录。

之后你在本地 Agent 里可以让它使用 `termark`：

```bash
termark assets list -q <keyword> --json
termark exec <asset-id> -- <command>
termark upload <asset-id> <local-path> <remote-path>
termark download <asset-id> <remote-path> <local-path>
```

这条路径适合“不想用 Termark 内置 Agent，但想让现有 Agent 操作 Termark 资产”的用户。

它的价值在于：Agent 不需要知道你的 SSH 密码、私钥、跳板机配置。Termark 已经保存了这些资产和凭证，外部 CLI 只是通过正在运行的 Termark 桌面端去访问受控能力。

能做的事情也比较明确：

- 搜索资产；
- 查看资产基础信息；
- 对指定资产执行一次性命令；
- 上传文件或目录；
- 下载文件或目录。

长任务也不应该让外部 CLI 一直挂着，应该在远端用 `tmux`、`nohup` 或 `systemd`。

这不是为了做一个万能 remote agent，而是给已有 Agent 一个安全、清晰、可复用的服务器入口。

## 三条路解决三类人

现在 Termark 的 AI 能力大概分成三条路。

第一条是内置 OpenAI 兼容 Agent。

适合想轻量接入、自主选择模型的人。配置 API、模型和确认策略，就可以在 SSH 终端旁边让 AI 看现场、分析问题、执行受控命令。

第二条是 Codex / Claude Code 接入。

适合已经信任这些 Agent 的人。Termark 负责提供当前 SSH 会话的 MCP 工具和安全边界，Codex / Claude 负责推理和执行计划。

第三条是外部 CLI。

适合本来就在本地使用 Codex、Claude、OpenCode 的人。你不用切换到 Termark 的 AI 面板，也能让本地 Agent 通过 `termark` 命令访问 Termark 管理的服务器资产。

我不想强迫用户只用一种方式。

有的人在乎接入门槛，有的人在乎模型能力，有的人在乎已有工作流。Termark 要做的是把这些路径统一到同一套资产、凭证、会话和安全策略上。

## 最后还是回到终端本身

我做 Termark AI 的时候，一直在避免一个方向：把终端变成一个看不见过程的自动化面板。

服务器工具最重要的不是“AI 看起来很忙”，而是用户知道它正在对哪台机器、哪个用户、哪个目录做什么。

所以我最后选择的形态很朴素：

AI 就站在当前终端旁边。

能看最近输出，能帮你分析问题，能把命令敲进当前终端，也能把工具结果整理回来。

但它不应该绕过你直接接管机器。

如果命令只是观察现场，就尽量少打扰；如果命令会改东西，就必须把要做什么摆出来，让用户决定；如果执行过程中需要密码、确认或临时判断，用户可以直接在终端里接手。

这也是我觉得 Termark 和很多“后台 Agent”不一样的地方。

它不是另开一个会话替你偷偷干活，而是绑定你正在看的终端。

人、终端、AI 在同一个现场里，这个边界清楚，服务器场景里才敢长期用。
